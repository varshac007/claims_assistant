Executive Summary

The Bloom Claims Assistant Portal is a modern, React-based intelligent claims management platform for life and annuity insurance. It replaces traditional ServiceNow UI with a high-performance "System of Engagement" layer, integrating directly with ServiceNow FSO as the System of Action.

The platform supports the full claims lifecycle — from First Notice of Loss (FNOL) intake through investigation, requirements gathering, beneficiary analysis, intelligent document processing, and payment authorization — with AI-powered insights embedded throughout. It is built exclusively on the DXC Halstack Design System and supports multi-persona workflows for Examiners, Supervisors, and Managers.

This scope document defines the features, phasing, integration requirements, effort estimates, assumptions, and risks for delivering the production-ready platform.

✗ Out of Scope
  • ServiceNow platform configuration and FSO setup
  • LexisNexis contract, credentials, or account setup
  • Azure Form Recognizer / AWS Textract provisioning
  • Document storage infrastructure (S3 / Azure Blob)
  • CI/CD pipeline build-out beyond Jenkinsfile scaffold
  • Native mobile application (iOS / Android)
  • Payment disbursement execution (portal authorizes; payment system executes)
  • Custom ServiceNow FSO table creation or schema changes
  • Third-party policy administration system integration
  • Claims litigation or legal hold management
  • End-user training delivery (train-the-trainer only)
  • Production infrastructure provisioning and sizing

Phase 3 — AI, IDP & Advanced Features
AI insights, anomaly detection, beneficiary analyzer, intelligent document processing, related policies, FastTrack routing
8–10 Weeks
Feature Description Priority Effort Notes
AI Insights Panel
 Compact card with expandable modal. Anomaly detection with HIGH / MEDIUM / LOW severity scoring, evidence-based recommendations, PASS/FAIL indicators, agential claim summary. Must Have 2 wks Requires agentic AI service in ServiceNow or external. Key differentiator.
Beneficiary Analyzer
 AI-powered conflict detection across beneficiaries. Flags competing claims, legal risks. Suggests resolution paths. Triggers workflow in ServiceNow. Must Have 1.5 wks Depends on beneficiaryAnalyzerService backend.
Intelligent Document Processing
 Auto-classify document type, OCR field extraction, confidence scoring, auto-populate claim fields from death certificates, IDs, and ACORD forms. Must Have 2 wks Azure Form Recognizer or AWS Textract. IDP service must be provisioned externally.
Related Policies Discovery
 Auto-search for other policies on deceased. Multi-policy grouping, total death benefit aggregation, warning alerts, quick-action to initiate related claims. Must Have 1 wk Significant revenue opportunity — surfaces unworked claims.
FastTrack / STP Routing
 Eligibility analysis engine. Determines claims suitable for straight-through processing. Bypasses non-essential requirements. FastTrack badge throughout UI. Must Have 1 wk Rules config in ServiceNow decision table.
LexisNexis Death Verification
 On FNOL submit, trigger LexisNexis death record verification. Return verification status to claim record. Should Have 1 wk Requires LexisNexis contract and credentials. Can stub for Phase 2.
Financial Calculators
 PMI Calculator (mortgage insurance, monthly payment, LTV) and Tax Withholding Calculator (gross benefit, federal/state tax, net benefit). Should Have 0.5 wk Pure frontend logic; no external integration.
Payment Quick View
 Disbursement summary panel showing payment status, method, and amounts. Authorization action (execution by external payment system). Should Have 0.5 wk Portal authorizes; does not execute payments.

Assumptions & Dependencies
Client Responsibilities
  • ServiceNow FSO instance provisioned with Insurance Data Model installed
  • ServiceNow OAuth 2.0 provider configured before Phase 1 start
  • LexisNexis contract and API credentials provided before Phase 3
  • IDP service (Azure Form Recognizer or AWS Textract) provisioned before Phase 3
  • Document storage (S3 or Azure Blob) provisioned and accessible
  • Agentic AI service architecture decision made before Phase 3 design
  • UAT resources available at end of each phase
Technical Assumptions
  • ServiceNow FSO Insurance Data Model tables used as-is (no custom schema changes)
  • React 18 + Vite + DXC Halstack 16 as mandated tech stack
  • Browser targets: Chrome, Edge, Safari (latest 2 versions)
  • No IE11 or legacy browser support required
  • Demo data layer maintained for all phases to support parallel sales demos
  • Role-based access controlled at ServiceNow; portal enforces UI-level only
Design & UX Assumptions
  • DXC Halstack Design System is the exclusive component library
  • No custom CSS frameworks or third-party component libraries except MUI for data grids
  • WCAG 2.1 AA accessibility compliance via Halstack baseline
  • Light and dark mode supported via Halstack theme
  • Bloom brand color (#5f249f purple) applied as primary; DXC cyan (#00AEEF) as accent
  • Responsive layout for desktop and tablet (portrait); mobile not in scope
Project & Delivery Assumptions
  • Phases are sequential; Phase N+1 begins after Phase N UAT sign-off
  • Weekly sprint cadence with fortnightly stakeholder demos
  • Change requests outside this scope will require formal scope change process
  • Training is train-the-trainer; end-user training delivery out of scope
  • Production infrastructure provisioning is client responsibility
  • Effort estimates assume 2–3 front-end developers + 1 tech lead
Risks & Mitigations
Risk Severity Probability Mitigation
ServiceNow FSO API access delayed — instance not provisioned or role access not granted in time for Phase 1 High Medium Demo data layer allows full UI development without live ServiceNow. API integration layered in once access is confirmed.
Agentic AI service architecture undefined — AI Insights Panel (Phase 3) blocked if backend service not selected High Medium Architectural decision required by end of Phase 2. Mock response layer allows UI development to proceed independently.
IDP accuracy below threshold — Azure Form Recognizer / Textract confidence on insurance documents may be low without model training Medium Medium Custom model training on sample documents prior to Phase 3 integration. Confidence thresholds displayed to examiners; manual override always available.
LexisNexis contract delay — Phase 3 death verification feature blocked without credentials Medium Low Stub mock service returns configurable verification status. Feature soft-launches when credentials are available with no code change.
Scope creep on AI features — AI Insights and Beneficiary Analyzer scope may expand during development Medium High AI features scoped to defined outputs (risk score, findings, recommendations). Additional AI capabilities logged as Phase 5 backlog items.
Halstack component gaps — edge case UI requirements not met by available Halstack components Low Low MUI is available as secondary library for data grids and specialized components. DXC Halstack team consulted for roadmap components.
