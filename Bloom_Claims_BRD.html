2 Executive Summary

Bloom Insurance currently manages life and annuity claims through the native ServiceNow interface, which was designed as a system of record rather than a system of engagement. Claims examiners navigate multiple disconnected screens, perform significant manual data entry, and rely on paper or scanned documents for death certificates and ACORD forms. This results in extended average processing times, elevated operational costs, inconsistent SLA compliance, and a degraded examiner experience that contributes to onboarding friction and error rates. The status quo presents a measurable risk to Bloom's service quality commitments and competitive positioning in the L&A market.

DXC Technology Smart Apps proposes to design and deliver the Bloom Claims Assistant Portal — a modern, React 18 + MUI v5 (Material UI) web application that sits as a System of Engagement over ServiceNow FSO (Financial Services Operations). The portal will provide a purpose-built claims workbench, guided FNOL intake wizard, intelligent document processing (IDP) with AI-driven auto-population, AI anomaly detection and risk scoring, related policy discovery, beneficiary conflict analysis, and a real-time manager analytics dashboard. All claim data and business logic remain authoritative within ServiceNow FSO; the React portal communicates exclusively through ServiceNow REST APIs secured by OAuth 2.0.

The expected business value of this engagement is material and measurable. Processing time reductions of 30% or more are targeted through FastTrack/STP routing for eligible claims, IDP-driven data capture, and guided workflow automation. Beneficiary conflict resolution, historically a source of delay and litigation risk, is targeted for a 50% cycle-time reduction via AI-assisted analysis. Real-time SLA dashboards will give supervisors and managers the visibility needed to intervene proactively, reducing SLA breach rates and improving Bloom's regulatory compliance posture. The investment directly supports Bloom's stated strategic objectives of operational efficiency, digital modernization, and superior policyholder experience.

3 Business Objectives
Reduce average claim processing time by 30% within 12 months of go-live through FastTrack routing, guided workflows, and IDP-driven data entry elimination.
Materially reduce manual data entry effort by leveraging Intelligent Document Processing (IDP) to auto-populate claim fields from uploaded death certificates, ACORD forms, and supporting documents.
Achieve FastTrack / Straight-Through Processing (STP) routing for 40% or more of eligible claims, reducing examiner touchpoints for low-complexity cases.
Achieve 100% SLA visibility across all open claims — providing examiners, supervisors, and managers with real-time on-track, at-risk, and overdue indicators.
Reduce beneficiary conflict resolution time by 50% through AI-powered conflict detection, severity scoring, and recommended resolution guidance surfaced inline on the workbench.
Provide real-time analytics and performance dashboards for management decision-making, including written premium trends, FastTrack rates, approval/decline ratios, and SLA compliance trends.

6 Technology Stack
Layer / Category Technology / Service Notes
Frontend Framework React 18, Vite, TypeScript Vite for fast development server and optimized production builds. TypeScript for type safety and maintainability.
UI Component Library MUI v5 (Material UI) Primary component library. Replaces DXC Halstack. All components sourced from MUI v5. Custom theming via MUI ThemeProvider.
State Management React Context API + Custom Hooks No Redux required for this scope. Context provides auth state, claim data, and UI preferences across component tree.
Authentication ServiceNow OAuth 2.0 / JWT Authorization Code flow. JWT tokens stored in memory (not localStorage). Refresh token rotation implemented.
Backend / System of Action ServiceNow FSO (Financial Services Operations) ServiceNow FSO Insurance Data Model. All claim data, workflow, and business rules authoritative in ServiceNow. Portal communicates via Table API and custom Scripted REST APIs.
Intelligent Document Processing Azure Form Recognizer or AWS Textract Client choice pending. Both support custom model training for insurance document types. Integration point abstracted behind a common IDP service interface.
Death Verification LexisNexis LexisNexis Accurint or InstantID Q&A for SSA-89 and death data. API contract required (BR-404). Called via a ServiceNow Integration Hub spoke.
Document Storage AWS S3 or Azure Blob Storage Client cloud preference determines provider. Signed URL pattern for secure document access. Documents indexed in ServiceNow Attachment table.
AI / Agentic Capabilities ServiceNow Now Assist or Claude API (TBD) Decision deferred to architecture phase. Now Assist preferred if SNow contract covers it; Claude API as alternative for agentic summarization and anomaly detection.
Deployment Static hosting (AWS S3 / Azure Blob) + CDN Portal deployed as a static SPA. CloudFront (AWS) or Azure CDN for global edge delivery. CI/CD pipeline via GitHub Actions or Jenkins.

8 Constraints & Assumptions
  • ServiceNow FSO Insurance Data Model must be installed, licensed, and configured in the target instance before Phase 1 API integration begins.
  • The portal UI will be built exclusively with MUI v5 (Material UI) components. No other CSS component frameworks (e.g., Bootstrap, Tailwind, DXC Halstack) will be introduced.
  • Custom CSS outside of the MUI ThemeProvider theming system is prohibited; all visual customization must be achieved through MUI's sx prop, styled() API, or theme overrides.
  • A valid LexisNexis API contract and production credentials must be in place and accessible before Phase 3 death verification integration is activated.
  • The client-selected IDP service (Azure Form Recognizer or AWS Textract) must be provisioned, credentialed, and accessible to the integration layer before Phase 3 document processing functionality is enabled.
  • Browser support is limited to modern browsers (Chrome, Edge, Safari — latest 2 major versions). Internet Explorer 11 is explicitly out of scope and will not be tested or supported.
  • A native mobile application (iOS or Android) is out of scope for this engagement. The portal will be responsive for tablet use but is not optimized for smartphone form factors.
  • All business logic, workflow automation, and data validation rules remain authoritative within ServiceNow. The React portal acts solely as a System of Engagement and does not replicate or shadow ServiceNow business logic.
  • Bloom Insurance will provide a dedicated non-production ServiceNow FSO instance for development and testing. Production data will not be used in any non-production environment.
9 Glossary
Term / AcronymDefinition
ACORDAssociation for Cooperative Operations Research and Development. ACORD forms are standardized insurance industry data exchange forms (e.g., ACORD 714 — Life Claim form).
BRDBusiness Requirements Document. A formal document capturing what a system must do to meet business objectives, written from the perspective of business stakeholders.
FastTrackA claim routing path that bypasses non-essential requirements for low-complexity, low-risk claims meeting pre-defined eligibility criteria, enabling accelerated processing and STP.
FNOLFirst Notice of Loss. The initial report made by a policyholder, beneficiary, or intake specialist to register a new claim event with the insurer.
FSOFinancial Services Operations. The ServiceNow industry module providing an insurance-specific data model, workflows, and pre-built integrations for life, annuity, and P&C insurance operations.
IDPIntelligent Document Processing. AI/ML-driven capability to automatically classify, extract, and validate data from unstructured documents (PDFs, images) using OCR and machine learning models.
IGOIn Good Order. A standard insurance term indicating that all required claim documents and information have been received and are complete, accurate, and sufficient for processing.
JWTJSON Web Token. A compact, URL-safe token format used to securely transmit authentication and authorization claims between the portal and ServiceNow APIs.
L&ALife & Annuity. The insurance product line encompassing term life, whole life, universal life, and annuity products. Primary product line for this portal.
LexisNexisLexisNexis Risk Solutions. A third-party data provider offering death verification services (Accurint, InstantID) used to validate insured death records against SSA and public death data.
MUIMaterial UI (MUI) v5. An open-source React component library implementing Google's Material Design system, used as the exclusive UI component library for this portal.
OAuth 2.0Open Authorization 2.0. An industry-standard authorization protocol used to grant the React portal delegated access to ServiceNow APIs on behalf of authenticated users.
OCROptical Character Recognition. Technology that converts images of text (scanned documents, photos) into machine-readable text, enabling IDP field extraction.
P&CProperty & Casualty. Insurance product line covering property damage, liability, and casualty risks. Referenced in analytics filtering but not the primary product line for this portal.
RBACRole-Based Access Control. A security model in which system access permissions are assigned to roles (Examiner, Supervisor, Manager, Intake) rather than individual users.
ServiceNow FSOSee FSO above. The authoritative system of record and system of action for all claim data, workflow, and business logic in this architecture.
SLAService Level Agreement. Contractual or regulatory time-bound commitments for claim processing milestones (e.g., acknowledgment within 3 business days, decision within 30 days).
SSNSocial Security Number. A US government-issued personal identification number. Classified as PII and subject to masking requirements under BR-1304.
STPStraight-Through Processing. Automated end-to-end claim processing with minimal or zero manual examiner intervention, enabled by FastTrack routing for eligible claims.
System of ActionThe authoritative backend system where business logic executes and data is persisted. In this architecture: ServiceNow FSO.
System of EngagementA purpose-built user interface layer designed for a specific user persona and workflow, providing a superior experience over the native System of Action UI. In this architecture: the React + MUI v5 portal.
